# 🚀 STM32H750双槽升级完整工作流程

## 📋 系统架构概述

### 核心特点
- **Bootloader**: 128KB内部Flash，负责升级管理和Application加载
- **Application**: 存储在QSPI Flash，加载到512KB RAM中执行
- **双槽机制**: Slot A/B，确保升级安全和回滚能力
- **组件化升级**: 支持Application、Web Resources、Config、ADC Mapping独立升级

### 存储器分配
```
内部Flash (128KB):
  0x08000000 - 0x0801FFFF: Bootloader

QSPI Flash (8MB):
  0x90000000 - 0x90010000: 升级元数据 (64KB)
  0x90010000 - 0x90110000: Application Slot A (1MB)
  0x90110000 - 0x90290000: Web Resources Slot A (1.5MB)
  0x90290000 - 0x902A0000: Config Slot A (64KB)
  0x902A0000 - 0x902B0000: ADC Mapping Slot A (64KB)
  0x902B0000 - 0x903B0000: Application Slot B (1MB)
  0x903B0000 - 0x904B0000: Web Resources Slot B (1.5MB)
  0x904B0000 - 0x904C0000: Config Slot B (64KB)
  0x904C0000 - 0x904D0000: ADC Mapping Slot B (64KB)
  0x904D0000 - 0x90800000: 未使用区域 (3.2MB)

AXI SRAM (512KB):
  0x24000000 - 0x2407FFFF: Application运行区
```

## 🎯 1. 正常发版流程

### 1.1 首次部署流程

#### Step 1: 系统初始化
```bash
# 1. 创建初始元数据和烧录bootloader
cd scripts
python init_system.py init

# 这会执行：
# - 编译bootloader和application
# - 创建初始升级元数据
# - 烧录bootloader到内部Flash
# - 烧录元数据到QSPI Flash
# - 烧录application到槽位A
```

#### Step 2: 验证部署
```bash
# 监控串口输出，验证系统正常启动
python dev_debug.py monitor COM3

# 期望看到的输出：
# [Bootloader] Starting...
# [Bootloader] Reading upgrade metadata...
# [Bootloader] Active slot: A
# [Bootloader] Loading application from slot A...
# [Bootloader] Jumping to application...
# [Application] Started successfully!
```

### 1.2 正常版本发布流程

#### Step 1: 准备发布版本
```bash
# 1. 更新版本号
# 在board_cfg.h中修改：
#define FIRMWARE_VERSION_MAJOR   1
#define FIRMWARE_VERSION_MINOR   0
#define FIRMWARE_VERSION_PATCH   2
#define FIRMWARE_VERSION_BUILD   1

# 2. 构建发布版本
cd release_scripts
python build_release.py --version 1.0.2.1

# 这会生成：
# - application_v1.0.2.1.bin (应用程序二进制)
# - web_resources_v1.0.2.1.bin (Web资源)
# - config_v1.0.2.1.bin (配置文件)
# - adc_mapping_v1.0.2.1.bin (ADC映射)
# - upgrade_package_v1.0.2.1.zip (完整升级包)
```

#### Step 2: 测试升级包
```bash
# 1. 在测试环境验证升级包
python test_upgrade.py upgrade_package_v1.0.2.1.zip

# 2. 模拟升级流程
# - 上传升级包到设备
# - 验证升级过程
# - 确认可以回滚到之前版本
```

#### Step 3: 发布到生产环境
```bash
# 1. 通过Web界面或串口命令上传升级包
# 2. 监控升级进度
# 3. 验证新版本功能
# 4. 如有问题，支持一键回滚
```

## ⚠️ 3. 重要约束和注意事项

### 3.1 Application必须在RAM中运行
- **原因**: STM32H750内部Flash只有128KB，已被Bootloader占满
- **实现**: Application存储在QSPI Flash，Bootloader将其加载到AXI SRAM中执行
- **限制**: Application大小不能超过512KB

### 3.2 调试限制
- **无法直接调试**: Application在RAM中运行，传统调试方法不适用
- **解决方案1**: 使用串口调试，通过printf输出调试信息
- **解决方案2**: 创建RAM调试版本，使用GDB调试
- **解决方案3**: 通过Bootloader提供的调试接口

### 3.3 升级安全
- **双槽机制**: 确保升级失败时可以回滚
- **CRC校验**: 每个组件都有CRC校验，确保数据完整性
- **重试机制**: 升级失败时自动重试，超过次数则回滚
- **断电保护**: 升级过程中断电也能恢复

## 🎯 5. 最佳实践

### 5.1 开发建议
1. **使用串口调试**: 在代码中添加适当的调试输出
2. **增量编译**: 使用watch模式，提高开发效率
3. **版本管理**: 每次提交都要更新版本号
4. **测试先行**: 在测试环境充分验证后再发布

### 5.2 升级建议
1. **分阶段升级**: 先在少量设备测试，再全量发布
2. **保留回滚**: 确保能快速回滚到之前版本
3. **监控升级**: 实时监控升级进度和成功率
4. **定期备份**: 定期备份重要配置和数据

### 5.3 故障排除
1. **串口日志**: 第一时间查看串口输出
2. **版本检查**: 确认当前运行的版本号
3. **元数据检查**: 检查升级元数据是否正确
4. **硬件检查**: 确认QSPI Flash和RAM工作正常

1. **检查串口输出**: `python scripts/dev_debug.py monitor COM3`
